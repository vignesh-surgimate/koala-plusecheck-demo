Timezone Strategy — everything on Eastern Time

🕓 Node side

One module (config/timezone.ts) sets process.env.TZ as a side effect and is the first import of every entrypoint. All generated dates — today(), tomorrow(), surgery schedules, report timestamps — mean the application's ET calendar day, regardless of runner locale. Proven on an IST machine: framework clock reports GMT-0400.

🌐 Browser side

Every context — config projects and manual contexts alike — receives timezoneId: 'America/New_York', so the app under test renders ET too. Generated dates and rendered dates can never disagree, even across the EST/EDT switch (IANA zone handles DST automatically).

Authentication Lifecycle

Login UI interaction lives in exactly one class (LoginPage); the configured role is mandatory, additional roles are best-effort — so one broken account never blocks the whole run.

Environment Management Strategy

One switch — PLAYWRIGHT_ENV — retargets six things at once. Environment knowledge lives in a single registry; nothing else hard-codes a URL or default.

URLs & projects

Base + API URLs resolve from the registry; BASE_URL can override ad-hoc.

Data & credentials

Every repository row and credential record is environment-scoped — staging tests use staging accounts only.

Runner & sessions & reports

<ENV>_TestData.json selection, per-env auth caches, and env-stamped reports — all from the same switch.

npm run test:preprod      # PLAYWRIGHT_ENV=pre-production
npm run test:prod         # production — automatically @smoke only
