Architecture · Flagship

Data-Driven Testing

Two pillars: the Runner Manager (external run control via SQLite → canonical JSON) and the module-dataset pattern — credentials backed by SQLite, never in JSON or env vars.

Runner Manager — the run is data, not code

Which tests run, in what priority, with what description, and how many data iterations — decided by one external record per test case in SQLite. QA leads reshape tonight's regression by editing the runmanager table; engineers commit nothing.

DB is the single credential backend — credentials never sit in JSON, env vars, or code. The IUserSource interface seam makes a vault or MySQL backend a one-class addition.

SQL → JSON Canonical Conversion

Every test run auto-regenerates canonical JSON from SQLite (default DATASOURCE=db). The runner always reads fresh JSON — no stale data, no manual sync.

Module-Dataset Pattern

Dataset keyed by module name + sequence number, scoped by sandbox + environment, with per-env snapshots stored by sandbox name.

// SQLite module tables → snapshot data/moduledata/<env>/<Sandbox>_TestData.json
const login = moduleData.getDataset('LoginTest', 2);  // "LoginTest_2".dataset
await loginPage.login(String(login.UserID), String(login.Password));

// write-back (InsertOutputDatatoDb port) — outputs feed later modules
moduleData.writeOutputData(
  'INSERT INTO outputdata (Sandbox, Environment, Module, OutputKey, OutputValue) VALUES (?,?,?,?,?)',
  [testConfig.client, testConfig.environment, 'AddPatientTest', 'PatientID', createdId],
