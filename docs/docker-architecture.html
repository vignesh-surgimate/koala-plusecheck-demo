Docker Execution Architecture

The whole stack — test runner, Allure service, Elasticsearch and Kibana — runs as one compose network. The execution image is built on Microsoft's official Playwright base, so browsers, fonts and system dependencies are pinned and reproducible.

Container Topology

The Execution Image — real Dockerfile

FROM mcr.microsoft.com/playwright:v1.48.0-noble

WORKDIR /app
RUN apt-get update && apt-get install -y sqlite3 && rm -rf /var/lib/apt/lists/*

# dependency layer cached separately from source for fast rebuilds
COPY package.json package-lock.json ./
RUN npm ci
COPY . .

CMD ["npx", "playwright", "test", "--project=staging-chrome", "--grep", "@smoke"]

Scaling Model

⚙ Workers (in-container)

4 parallel workers per container in CI (auto-detected locally). Each worker owns an isolated browser context — no shared state, no cross-talk.

🔀 Shards (across containers)

--shard=1/4 … 4/4 splits the suite across machines or container replicas. 4 workers × 4 shards = 16× parallel capacity on the regression tier.

🏥 Matrix legs (across tenants)

Each of the 20 sandbox × role legs is an independent container invocation — fan out the whole customer base on a Kubernetes job queue or CI matrix.

Operate It

# build the execution image
docker build -f docker/Dockerfile -t surgimate-automation .

# full observability stack: runner + allure(:5050) + elasticsearch(:9200) + kibana(:5601)
docker compose -f docker/docker-compose.yml up

# one tenant leg, parameterized like any local run
docker run -e PLAYWRIGHT_CLIENT=eCW -e PLAYWRIGHT_ROLE=Surgeon surgimate-automation

# sharded fan-out (per CI machine / replica)
docker run surgimate-automation npx playwright test --grep @regression --shard=2/4
Why containers matter here: identical browser binaries between a tester's laptop and CI eliminates "works on my machine" flake triage; volumes persist results/Allure history across runs; and the ELK pair turns every execution into queryable telemetry without any external SaaS dependency.
